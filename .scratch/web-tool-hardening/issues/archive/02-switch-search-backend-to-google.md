# Switch web search from DuckDuckGo scraping to Google

## Context

Search is currently DuckDuckGo, hardcoded in three places in
`agent-harness/src/marketing_os/adapters/tools/websearch_playwright.py`:

- the URL template default `https://duckduckgo.com/html/?q={query}` (line 118),
- the result parsing selectors `.result`, `a.result__a`, `.result__snippet`
  (`_search_on_worker`, lines 208–217),
- the DDG redirect-link decoder `_decode_result_href` (lines 44–63).

So this is not a one-line URL swap: the parsing layer is engine-specific.

## Options

1. **Google Custom Search JSON API (recommended).** Official, stable JSON, no
   scraping or browser needed for search (Playwright stays for `web_fetch`).
   Implement as a new `WebSearchTool` backend (plain HTTP), keyed off settings.
   Requires a Google API key + Programmable Search Engine ID; free tier is 100
   queries/day, then $5/1000.
2. **Scrape google.com/search with Playwright.** No API key, but headless
   Chromium hits consent walls and CAPTCHA/bot detection quickly, and Google's
   result markup changes often. Fragile; not recommended.
3. **A paid SERP provider (SerpAPI, Serper, Brave Search API).** Stable JSON,
   simpler than CSE setup, but a new paid dependency.

## Decisions (resolved in triage — see Triage Notes below)

- **Option 1 (Google CSE) rejected** — deprecated / deprecating soon.
- **Option 2 chosen**: scrape `google.com/search` via Playwright — maximum
  search freedom, and the agent can navigate found sites via the existing
  `WebFetch` tool.
- **The ports-and-adapters seam already exists** (`WebSearchTool` ABC in
  `websearch.py`). This issue adds a new adapter + a backend _selector_; it does
  not re-architect the seam.
- **Selection model = fallback chain.** Config names backends in priority order
  (e.g. `google,serpapi`); the resolver tries the first, and on a recoverable
  failure or empty result, falls through to the next. This is what "use either
  one or both based on configs" means operationally.
- **Scope of this issue = Google adapter + selector only.** SerpAPI is a future
  adapter in a separate issue that drops into the same seam and slots into the
  fallback chain. Do **not** build SerpAPI here.

## Agent Brief

**Category:** enhancement
**Summary:** Add a Google-backed `WebSearchTool` adapter and a config-driven,
priority-ordered backend resolver, replacing the current single hardcoded
DuckDuckGo-scraping backend.

**Current behavior:**
Web access is a pluggable `WebSearchTool` port (`search` / `fetch` / `close`)
with two adapters: a no-op default and a Playwright/Chromium backend that scrapes
**DuckDuckGo's** HTML endpoint (engine-specific URL template, result selectors,
and a DDG redirect-link decoder). Backend selection is binary: a single
resolver gates on `enable_web` (`MARKETING_OS_WEB=1`) and returns one hardcoded
Playwright backend or `None` (→ no-op). There is no way to choose _which_ search
engine, and no fallback if a backend fails.

**Desired behavior:**

- A new adapter performs web **search** against `google.com/search` via
  Playwright, parsing Google's result markup into the same source-attributed
  result format the existing formatter produces. Page **fetch** continues to use
  the Playwright browser (reuse the existing fetch path; don't duplicate it).
- The resolver reads an **ordered list of backend identifiers** from config and
  builds a **fallback chain**: `search()` tries the first backend, and on a
  recoverable failure (a `ToolError`) or an empty result set, falls through to
  the next; the last backend's failure surfaces as a `ToolError` to the
  specialist (consistent with issue 01). A single configured backend behaves
  exactly as it does today.
- Google-specific failure modes (consent interstitial, CAPTCHA / bot-detection
  redirect, markup-parse yielding zero results) are treated as recoverable —
  they raise `ToolError` (feeding the fallback), never a raw Playwright error
  and never a fatal run crash.
- `NoopWebSearch` and the existing DuckDuckGo Playwright backend remain
  available and selectable by identifier; the DuckDuckGo behavior is not
  deleted.

**Key interfaces:**

- `WebSearchTool` (the port) — unchanged; the new backend implements it.
- New Google search adapter — implements `search`; delegates/ reuses the
  Playwright `fetch` and browser lifecycle rather than reimplementing it.
- Backend **resolver** (currently `_resolve_web_backend`) — change from
  returning a single backend to composing a priority-ordered fallback chain
  from config. The chain itself should be expressible as a `WebSearchTool` so
  the rest of the graph wiring is untouched.
- **Config shape** — add a backend _selector_ to `Settings` (e.g. an ordered,
  comma-separated `MARKETING_OS_WEB_BACKENDS` naming `google` / `duckduckgo` /
  `noop`), read via `config.py` per repo standards. `MARKETING_OS_WEB` remains
  the on/off gate; the new setting picks and orders backends when web is on.
  Pick sensible defaults and document them.
